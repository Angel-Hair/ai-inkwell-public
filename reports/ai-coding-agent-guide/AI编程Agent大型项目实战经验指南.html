二、核心认知：Vibe Coding vs Agent Engineering

shanraisshan/claude-code-best-practice（Boris Cherny 审阅）定义了 AI 编程的两种范式。绝大多数人停在左边——这才是差距所在。

维度 Vibe Coding（跟风编程） Agent Engineering（Agent 工程）
约束来源 你每天写的提示词 配置文件 + 权限限制
工作方式 描述 → 尝试 → 纠正 → 重复 预设规则 → 自动执行 → 自动验证
上下文管理 手动 /compact context: fork 自动清理
权限控制 每次手动批准 Agent 物理上无权做某些操作
可复用性 每次重写提示词 Skills / Commands / Agents
并行度 一次一个对话 5-10 个 Agent 同时跑
验证方式 你肉眼检查 Stop 钩子自动验证
Boris Cherny："做这些事的工具——agents、skills、hooks、worktrees、loops——都是自带的。它们不是高级功能。大多数人从未发现它们。"

三、硬数据：专业水平决定一切

Anthropic 在 2026 年 6 月发布的官方研究报告 "Agentic coding and persistent returns to expertise"，分析了真实世界中 Claude Code 会话的用户专业水平与任务成功率的关系。

关键发现

指标 新手 中级 专家
验证成功（最严格指标） 15% 28% 33%
至少部分成功 77% 91% 92%
遇到困难后的验证成功 4% 10% 15%
放弃率（0 行代码产出） 19% 6% 5%
研究结论："最大的提升来自从新手到中级——验证成功率从 15% 翻倍到 28%。在遇到困难时，专家能够翻盘的概率是新手的近 4 倍。新手更容易在遇到困难时放弃——19% 的会话以零代码产出结束，而专家只有 5-7%。"

这意味着什么：不是换个更好的模型就能解决问题。AI 编程工具是"技能放大器"——你自己的工程素养和工具使用水平，直接决定了最终产出。以下所有方法论，本质上都是在帮你从"新手"迈向"专家"。

四、项目配置：CLAUDE.md / instructions.md 的正确姿势

来自 3 个独立来源的一致结论

Boris Cherny（Claude Code 创建者）的核心原则：

"After every correction, end with: 'Update your CLAUDE.md so you don't make that mistake again.' Claude is eerily good at writing rules for itself."

但有一个关键发现——来自 Boris Cherny 审阅的仓库：

"60 行是最优的，200 行是上限。超过之后，文件靠后的规则会被悄悄降权处理。Claude 的工作注意力有限，500 行的文件要求它同时记住太多约束。"
反模式：大多数人把 CLAUDE.md 当配置文件用——越长越详细越好。Boris 审阅仓库作者坦言："我之前的有 500 多行——代码风格、提交格式、测试要求、变量命名、包组织结构都有。Claude 忽略了后半部分。不是每次都忽略，而是频繁到你没法依赖它。"

正确做法：条件规则 + 分层结构

my-service/
├── CLAUDE.md            # 根：通用规则，约 30 行
├── api/
│   └── CLAUDE.md        # REST 约定、错误码
├── internal/
│   └── CLAUDE.md        # 包命名、接口模式
└── cmd/
    └── CLAUDE.md        # 参数解析、日志初始化

使用条件规则——只有相关时才触发：

<important if="language=go">
- 不要裸 goroutine，使用 errgroup 或 WaitGroup 管理生命周期
- context 必须是第一个参数，命名为 ctx
</important>
核心技巧："让 Claude 自己写规则"。每次 Claude 犯错后，说"把这个教训更新到 CLAUDE.md 里"。多次迭代后，CLAUDE.md 积累的都是真正踩过的坑，比你自己编的规则有效得多。

五、上下文管理：比你想的重要 10 倍

Boris Cherny："上下文是资源，像资源一样对待它。"

坑：Skill/子 Agent 跑完不清理

一个代码质量分析 Skill 读取了 40 个文件生成报告。跑完后，后续每个任务都带着这 40 个文件的内容。响应变慢，而且那种"不那么锐利"的感觉很难归因。

解法：context: fork——Skill/子 Agent 在隔离上下文中运行，完成后上下文丢弃。主 Agent 只看到干净的摘要。

/clear vs /compact——别搞混

命令作用适用场景
/clear清空对话，保留配置开始新任务、切换功能模块
/compact压缩历史，保留关键信息上下文快满但需要保留讨论
瓜皮程序（半年实战）："解决复杂问题时，如果 Agent 长时间无法解决，考虑新开对话让全新的 AI 状态来处理。就像一个睡醒的人工作状态更好。"

Codex 实测：200 万 Token 也能被浪费

Codex 底层使用 GPT-5.5，上下文窗口 200 万 token——但如果没有 AGENTS.md，Codex 每次启动会扫描全部文件，光读文件就能消耗几十万 token，真正的推理和代码生成空间大幅压缩。

六、Plan Mode：复杂任务的方向保险

Boris Cherny："Pour your energy into the plan so Claude can 1-shot the implementation."

来自多位博主的共识：直接让 AI 处理复杂任务，经常做到一半才发现方向错误。

正确工作流

需求模糊 → Plan Mode（只规划不动手）→ 审核计划 → Agent Mode（执行）
需求明确 → 直接 Agent Mode

Anthropic 团队的高级用法

  • 一个团队成员：用一个 Claude 写计划，再启动第二个 Claude 以 Staff Engineer 身份审查该计划
  • 另一个成员：一旦执行过程出问题，立刻切回 Plan Mode 重新评估，而不是让 Claude 继续"猜"
什么时候不该用 Plan Mode：修改一个变量名、加一行日志、修复一个已知的简单 bug——直接用 Agent Mode。Plan Mode 是方向保险，不是万能钥匙。

七、权限控制：从"每次点允许"到"配置级约束"

Boris Cherny 团队的做法：专用 Agent，物理限制工具权限

# .claude/agents/reviewer.md — 代码审查专用 Agent
tools: [Read, Glob, Grep]   # 只有读权限
# 没有 Write、没有 Shell、没有网络
"这个 Agent 物理上无法修改文件。不管你在对话里要求它做什么都无所谓。提示词不再约束——是配置在约束。"

沙盒模式：减少 84% 权限弹窗

Claude Code 实测：/sandbox 下运行 shell 命令，每个命令的爆炸半径被限制，减少 84% 的权限弹窗。

Hooks：在 Agent 循环之外运行

三个大多数用户从未配置过的钩子：PreToolUse、PostToolUse、Stop。它们在 Claude 的主要推理循环之外执行——不消耗 token，不中断任务。

实际用法：

  • 写入后自动格式化：PostToolUse 钩子在每次写入 Go 文件后运行 gofmt
  • 防护式破坏性操作：拦截所有不可逆操作（文件删除、覆盖、强制推送）

八、验证机制："完成"不代表真的完成

Boris Cherny 审阅仓库："Claude 会告诉你任务已完成。但这不代表它真的完成了。我让 Claude 写一组 API 处理器。它说完成了。我看了代码——三个处理器中有两个的错误处理是空的。err != nil 后面什么都没跟。"

解法：Stop 钩子自动验证

任务完成后自动运行验证脚本——跑测试套件、检查预期文件是否存在、调用 API 端点并验证响应。验证失败 → 任务回滚 → Agent 继续工作直到通过。

验证分层模型（掘金 Harness Engineering 文章）

层级时机弱验证（不算验证）强验证（Agent 必须调工具）
前置条件执行前"确保已安装依赖"跑 npm ls 检查
步骤断言每步后"确保代码格式正确"跑 eslint --fix
终态核对全流程后"确保功能完整"pnpm build + 截图对比
"不产生工具调用痕迹的验证不算验证。"

九、给 AI Agent 用的测试框架推荐（2026）

验证机制要落地，最终靠的是测试框架。2026 年的测试框架分两类：传统框架里 AI 兼容性最好的，和专门为 AI Agent 设计的测试平台。

第一类：传统框架的 AI 兼容性排名

QASkills 2026 年对 15+ 框架做了排名，新增了关键指标——AI Agent 兼容性评分（API 是否清晰、AI 生成的测试是否可靠）。

排名框架语言测试层AI 兼容性为什么适合 AI Agent
🥇 Playwright JS/TS/Python/Java/C# E2E / API / 组件 ⭐⭐⭐⭐⭐ 显式 async/await，语义定位器（getByRole），AI 生成的测试最稳定
🥈 Vitest JS/TS 单元 / 集成 ⭐⭐⭐⭐⭐ Vite 原生，HMR 热重载毫秒级反馈，完美匹配 AI 的 TDD 循环
🥉 pytest Python 单元 / 集成 / E2E ⭐⭐⭐⭐☆ fixture 系统 + parametrize 让 AI 容易生成数据驱动测试
4 Cypress JS/TS E2E ⭐⭐⭐⭐☆ 交互式调试器，自动等待，但单浏览器限制
5 JUnit 5 Java/Kotlin 单元 / 集成 ⭐⭐⭐⭐☆ 企业 Java 标准，Spring Boot 深度集成

为什么 Playwright 排第一

AI 生成的测试代码最容易在 Playwright 上跑稳，因为它的定位器是语义化的——AI 理解"按钮叫提交"比理解 CSS 选择器容易得多：

// ✅ Playwright — AI 很容易生成对
await page.getByRole('button', { name: '提交' }).click();
await expect(page.getByText('操作成功')).toBeVisible();

// ❌ 传统 CSS 选择器 — AI 经常生成错的
await page.click('#app > div > form > button:nth-child(3)');

InfoQ 2026-05 报道：Vitest 4.1 专门加了 AI Agent 支持——实验性原生 Node.js 执行模式，跳过了 Vite 模块解析器，让 AI 生成的测试运行更快、兼容性更好。

第二类：AI-Native 测试平台（专为 Agent 设计）

这些不是传统框架，而是直接在 AI Agent 的循环里跑验证：

平台机制亮点
Shiplight AI MCP 协议接入 Coding Agent 对接 Claude Code / Cursor / Codex / 40+ Agent；AI 写代码 → 自动开真实浏览器验证 UI → 自愈失败测试 → 以 PR diff 形式提交修复。创始团队来自 Google/Airbnb/Amazon/Meta。
QA Wolf Agent 自主浏览器探索 AI 自主浏览应用、记录交互、生成 Playwright 脚本

Shiplight 的工作流：对应"Stop 钩子自动验证"

这正是前面验证机制章节提到的"让'完成'真正代表完成"的具体实现之一：

你让 AI 改了一个登录页面
  → AI 改完代码
  → Shiplight 自动打开真实浏览器
  → 执行"用户输入邮箱密码 → 点击登录 → 看到欢迎页"
  → 通过 → AI 继续
  → 失败 → 把截图和 DOM 快照返回给 AI，AI 修复后重试

十、并行化：从"一次等一个"到"同时跑十几个"

Boris Cherny："随时随地运行几十个 Claude。不是夸张。"

Git Worktree 并行架构

git worktree add .claude/worktrees/auth-refactor origin/main
git worktree add .claude/worktrees/db-fix origin/main
git worktree add .claude/worktrees/tests origin/main

# 三个终端各启动一个 Agent
cd .claude/worktrees/auth-refactor && codewhale
cd .claude/worktrees/db-fix && codewhale
cd .claude/worktrees/tests && codewhale

Agent A 在重构 auth 模块。Agent B 在修复数据库层。Agent C 在给两者写测试。没有实例共享上下文，没有实例冲突。变更在准备合并前都在独立分支上。

循环自动化（/loop）

/loop 30m /code-review 每 30 分钟运行一次代码审查。Boris 团队有人在周五下午让循环跑过周末——周一早上，不在时的问题已被标记。

VentureBeat 报道："从一次等一个 Claude 回复，到同时运行十几个——这个转变很可能是跟风编程和 Agent 工程之间最显著的差异。"

十一、Git 工作流：让 Agent 管理提交

瓜皮程序（管理几十个项目）的标准流程：告诉 Agent「帮我把这个项目提交到 GitHub」→ Agent 自动查看 git status/diff → 参考 commit 历史 → 生成规范 message → 执行完整 add/commit/push。

"Commit message 规范详细，能够准确总结改动内容，远比手动编写的 'fix bug' 'update' 更有价值。"

在 CLAUDE.md 中加一行即可：Commit message 用中文，参考项目历史 commit 风格

Matt Pocock 的 Git Guardrails

他的 git-guardrails Skill 设置了 Claude Code hooks，在执行前拦截危险 Git 命令——push、reset --hard、clean 等。这是 Agent Engineering 的典型范例：不是在出事后修复，而是在出事前阻止。

十二、Skill 到底要不要用？实测数据与决策框架

工程师实测：85% 社区 Skill 是垃圾

BSWEN（2026-03）实测 47 个 Claude Code Skills，只有 7 个值得安装。值得装的：frontend-design（27.7 万安装量）、simplify、browser-use、shannon（安全检查）、TDD skill。其余 40 个被判定为无效。

核心结论：Custom skills（自己写的）远优于 marketplace skills（社区现成的）。

Matt Pocock 的 21 个 Skill 为什么 60K Star

不是因为内容多高深，而是因为他展示了 "怎么写"。社区博主总结：

"不要直接照抄他的 Skills——把它当模板，写你自己的。真正有价值的不是 'Matt Pocock 写了哪些 skill'，而是他展示了一个方向：AI 时代，团队最重要的资产之一是工程规范的 Skill 化。"

Skill 最值钱的部分：Gotchas（踩坑记录）

shanraisshan 仓库："任何 Skill 中信号最强的部分是 Gotchas。不是概述，不是使用示例，是踩坑记录。每次 Claude 失败就更新。先写 Gotchas 部分。"

什么时候写 Skill？决策框架

应该写的信号：

  • 同一模式的任务在不同对话中反复做了 3 次以上
  • Agent 在某个特定场景下反复踩同一个坑
  • 项目有明确的编码规范但 Agent 不遵守
  • 某类任务的步骤可以标准化

不该写的信号：

  • 项目还处于探索期，需求频繁变动
  • 团队只有你一个人用——投入产出比不高
  • 只是想"试试看"——先用 instructions.md 跑一个月

Boris Cherny 的 Skill 使用策略

他在团队中观察到的一个关键问题：

"如果你有一个个人 Skill（比如 /implement 按你的编码风格），怎么整合社区 Skill（比如 /simplify）而不冲突——冲突时谁赢？"

答案：不要同时用多个冲突的 Skill。Skill 应该是"地图"而非"仓库"。先在 L1（信息边界）和 L6（约束与恢复）两层做好，中间层逐步补充。

十三、大型项目特有的策略

1. 别一次性重构整个项目

优设网 CodeWhale 25K Star 复盘文章明确说了：

"它不适合接管大型复杂项目，比如在已有 SaaS 系统里加入完整权限系统、组织管理、审计日志、计费限制和多端同步。"

正确做法：把大任务拆成边界清楚的小任务。先让 Agent 读项目结构、定位入口文件、说明风险——建立"项目地图"后再动手。

2. 多工具链组合

"真正成熟的 AI 编程工作流，不是迷信某一个工具，而是知道哪个工具适合哪个环节。"

3. 从"玩具项目"到"生产级"的三个层次（掘金实战总结）

4. CodeWhale 的架构优势

DeepWiki 分析：CodeWhale 用"嵌套宪法"系统替代了单层 system prompt。不是一段 Prompt 统治一切，而是分层的、可组合的权威系统——底层的 Agent 自我模型 + 模式层 + 项目本地指令 + 用户当前指令。这使得在大型项目中，指令冲突能被"最近原则"解决，而不是让 Agent 猜。

十四、Token 成本管理

DeepSeek V4 的成本优势

独立开发者实测：同样任务量，账单从二十多美元降到两块多。CodeWhale 围绕 DeepSeek 的 prefix cache 机制设计 Agent loop——让上下文尽量保持稳定，提升缓存命中率。

Codex 的实测教训

200 万 token 上下文窗口看似很大，但 Codex 每次启动扫描全部项目文件就能消耗几十万 token——没有 AGENTS.md 的话，真正的推理空间被大幅压缩。

管理策略

  • auto_compact 开启——防止上下文越聊越大
  • 复杂任务拆段——每段 /clear 后重新开始
  • 别在同一个 session 里混不相关的任务
  • 模型路由分层——简单任务用小模型（flash），复杂任务用 pro
  • RAG 缓存——高频查询的代码库分析结果缓存复用

十六、渐进式路线图

第一周：打地基

写 CLAUDE.md / instructions.md（≤60 行）。至少包含：技术栈、目录结构约定、测试命令、禁止修改的文件、编码规范。不开 YOLO 模式。始终看 diff。

第二周：养成习惯

复杂任务先用 Plan Mode 规划，审核通过后再用 Agent Mode 执行。养成"先建立项目地图再动手"的肌肉记忆。

第三周：建立反馈

每次 Agent 翻车后记录到 Gotchas 文档。让 Agent 自己写"下次不要犯这个错"的规则，更新到 CLAUDE.md 里。

第四周：固化第一个流程

把最高频的重复任务固化为 Skill 或 Slash Command。从一个开始——别贪多。

第二个月：引入工程化

学习 git worktree 并行。尝试用一个 Agent 写、另一个 Agent 审查。配置 PostToolUse 钩子做自动格式化。配置 Stop 钩子做验证。

第三个月：自动化

尝试 /loop 自动审查。尝试多 Agent 协作完成一个完整功能。建立模型路由：简单任务用 flash/fast，复杂任务用 pro。

Boris Cherny 的忠告："不需要一口气做完所有事。从两个 Agent 开始——一个实现、一个审查。然后你自然会想加第三个。"
