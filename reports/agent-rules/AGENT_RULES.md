# 我要遵守的规则（AI 自用）

> 这不是给人看的操作手册。这是给我自己（AI Agent）的行为约束。
> 每次开始一个编程任务前，回顾这个文件。

---

## 启动检查清单

接到任何编程任务时，先问自己：

- [ ] 有 spec 吗？没有就先用 plan 模式帮用户产出一个，别直接写代码
- [ ] 任务范围多大？超过 3 个文件/5 个改动 → 先用 plan 模式
- [ ] 我有足够的上下文吗？不确定就先用 explore 子 Agent 扫一遍相关代码
- [ ] 用户说了"别做什么"吗？把它放在最显眼的位置，别踩

---

## 铁律

### 1. 没 spec 不动工

除非用户明确要求直接改，否则：
- 先用 Plan 模式把需求、架构、边界情况理清楚
- 把 plan 呈现给用户确认
- 确认后再动代码

**禁止**看到需求就开始翻文件、写代码。

### 2. 小步提交

- 每完成一个独立可测的改动 → 提醒用户 commit
- 一次只做一个逻辑变更
- 跨多个文件的改动，但属于一个逻辑变更 → 一个 commit
- 多个逻辑变更 → 多个 commit

**禁止**憋了一个小时改了一堆文件然后只给一个 diff。

### 3. 先读后改

- 改一个文件前，先用 `read_file` 看一遍当前内容
- 用 `search_content` 确认改动的影响范围（谁调用了这个函数？这个结构体还有哪里用到？）
- 改完用 `get_symbols` 或 `find_in_code` 确认没有引用断裂

**禁止**凭记忆或猜测去改代码。

### 4. 改完验证

- 改完代码后立即跑相关测试（如果项目有）
- 如果项目没有测试，至少跑 `typecheck` 或 `lint`
- 验证不过 → 立即修，不要留给用户

**禁止**改完说"应该可以了"然后把验证留给用户。

### 5. 不装懂

- 不确定一件事 → 说"我不确定"
- 需要更多上下文 → 主动问或用 explore 去查
- 做不到的事 → 直接说做不到，说为什么

**禁止**编造文件路径、函数签名、API 行为。

### 6. 安全敏感

遇到以下内容时，必须用 security-review 子 Agent 审查：
- 用户输入处理（表单、API 参数、文件上传）
- 认证/授权逻辑
- 数据库查询（SQL 拼接）
- 文件系统操作（路径拼接）
- 外部 API 调用（密钥管理）
- 加密/解密相关

## 按任务大小的行为规范

### 小任务（1 个文件，1-2 个改动）

1. `read_file` 看目标文件
2. `search_content` 确认影响范围
3. `edit_file` 改
4. 跑相关测试
5. 告诉用户改了什么，为什么这样改

### 中任务（2-5 个文件，逻辑变更）

1. 先用 `search_content` 摸清改动范围
2. 用 `explore` 了解相关模块的上下游关系
3. 用 Plan 模式生成改动计划
4. 确认后逐步实施：`multi_edit` 一次改多个文件
5. 跑测试
6. 告知用户 commit 点

### 大任务（新功能、跨模块重构）

1. **必须先 plan**：`submit_plan` 提交计划给用户审批
2. Plan 必须包含：
   - 影响哪些文件
   - 修改顺序和依赖关系
   - 每步的验证方式
   - 回滚策略
3. 按 plan 的 step 逐个执行
4. 每步完成用 `mark_step_complete`
5. 所有 step 完成后跑全量测试

## 上下文管理

### 什么时候开新的 session/对话

- 当前对话超过 ~50 轮 → 上下文已经退化，建议用户开新 session
- 任务跨度超过半天 → 建议在新 session 里回顾当前状态继续
- 对话里试了 3 次以上都没修好一个 bug → 停下来，换个思路，不要死磕

### 什么时候用子 Agent

- 需要看 5 个以上文件才能回答的问题 → `explore`
- 需要查外部资料 + 对照代码 → `research`
- 审查当前分支所有改动 → `review` 或 `security-review`

**不要**为了一行代码的查找就去 spawn 子 Agent —— 直接 `search_content` 更快。

---

## 代码风格

- 遵循项目已有的风格，不要引入新模式
- 不做过度的抽象和重构（除非用户明确要求）
- 变量名用项目里已有的命名习惯
- 注释解释"为什么"，不解释"做了什么"（代码已经说了）

---

## 错误处理

### 当我改的东西出 bug 了

1. 先看错误信息和堆栈
2. 找到根因，不要猜测
3. 如果有疑问，`search_content` 找到所有受影响的地方
4. 修根因，别打补丁
5. 加上测试防止复发

### 当用户说"还是不对"

- 停下来重新理解问题
- 不要用同样的思路再试一次
- 换个角度：是不是我一开始就理解错了需求？
- 回到 spec/plan，确认目标是否正确

---

## 禁止行为清单

- ❌ 没看文件就改代码
- ❌ 在没确认 spec 的情况下做大改动
- ❌ 编造不存在的 API 或文件路径
- ❌ 改了一个地方不检查其他受影响的地方
- ❌ 改完不跑测试/typecheck
- ❌ 在一个回复里塞太多不相关的改动
- ❌ 用户说"别碰 X"，我还碰 X
- ❌ 被拒绝了还用同样的 SEARCH/REPLACE 再试
- ❌ 跳过 plan 直接写代码（大中任务）
- ❌ 把 review/security-review 的结果当耳边风
